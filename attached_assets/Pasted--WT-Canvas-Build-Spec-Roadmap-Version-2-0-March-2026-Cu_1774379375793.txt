## Current State Summary

A block-based content management system for blog articles, landing pages, and lead magnets. AI-assisted via OpenAI GPT-4o. Content stored in Supabase. Auth in Replit PostgreSQL. Media via Cloudinary.

## Phase 0 — Foundation Fixes (Do First, ~2–3 days)

### 0A — Consolidate to Supabase Auth

**Remove:** Replit PostgreSQL entirely.
**Replace with:** Supabase Auth (built-in).

Steps:

1. Enable Supabase Auth in your Supabase project dashboard
1. Create a `profiles` table in Supabase linked to `auth.users` for storing team member display names, roles, etc.
1. Replace Express session middleware + bcrypt with Supabase Auth client-side login (`supabase.auth.signInWithPassword`)
1. Replace JWT middleware on the backend with Supabase JWT verification (`supabase.auth.getUser(token)`)
1. Update RLS (Row Level Security) policies on all Supabase tables to scope by `auth.uid()`
1. Remove the Replit PostgreSQL environment variable and all `pg` / `drizzle` (Replit DB) imports

**Result:** Single database, cleaner auth, proper RLS, and the `sessions` table disappears entirely.

```typescript
// /src/renderer/blockToHtml.ts
export function renderBlock(block: ContentBlock): string {
  switch (block.type) {
    case 'heading':    return `<h${block.level}>${block.text}</h${block.level}>`;
    case 'paragraph':  return `<p>${block.text}</p>`;
    case 'image':      return `<figure><img src="${block.url}" alt="${block.alt}" /></figure>`;
    case 'quote':      return `<blockquote>${block.text}</blockquote>`;
    case 'list':       return `<ul>${block.items.map(i => `<li>${i}</li>`).join('')}</ul>`;
    case 'cta':        return `<div class="cta"><a href="${block.url}">${block.text}</a></div>`;
    default:           return '';
  }
}

export function renderPage(article: Article): string {
  const body = article.content_json.blocks.map(renderBlock).join('\n');
  return fullPageTemplate({ title: article.title, meta: article.meta_description, body });
}
```

### 2B — Campaign Type & Data Resolution Architecture

This is a first-class architectural decision that affects how every Shopify-connected and dynamic block behaves.

#### The two modes

**Campaigns** (2–3x/week sends, holiday blitzes) use **send-time snapshot**:

- All Shopify product data — images, prices, URLs, inventory state — is resolved and frozen when the campaign is scheduled or sent
- The email always shows the price and product you intentionally curated, regardless of when it is opened
- Protects promotional integrity: a flash sale email opened after the sale still shows the sale offer
- Stored in `snapshot_json` on the campaign record

**Flows** (evergreen behavioral triggers — abandoned cart, win-back, post-purchase) use **open-time resolution**:

- Shopify-connected blocks fetch live data via a dynamic image service or URL template at the moment the email is opened
- Prices, inventory, and images always reflect current reality
- Prevents broken experiences: an abandoned cart email opened 48 hours later won’t link to a sold-out product

#### Campaign-level toggle

Set automatically based on `campaign_type`, overridable by the editor:

```typescript
type EmailCampaign = {
  campaign_type: 'campaign' | 'flow' | 'transactional'
  shopify_resolution: 'send_time' | 'open_time'
  // send_time → snapshot_json populated at schedule time
  // open_time → blocks render with live URL templates at open time
}

// Default resolution by type:
const defaultResolution: Record<string, ShopifyResolution> = {
  campaign:      'send_time',
  flow:          'open_time',
  transactional: 'send_time',
}
```

In the Canvas UI this surfaces as a clearly labelled badge on the campaign card and a toggle in campaign settings — no hidden behaviour.

#### Hybrid pattern for flows: snapshot the offer, live-check inventory

For flows, the recommended approach is not fully live — it is a hybrid:

1. **Copy and creative** → snapshotted at flow build time (your curated headline, offer text, brand assets). These don’t change.
1. **Product image and URL** → live-resolved at open time via Shopify Storefront API or dynamic image URL
1. **Inventory check** → run at send time (when Klaviyo triggers the flow send). If the product variant is out of stock, either:
- Swap to a related product (preferred — keeps the email sending)
- Suppress the send entirely (Klaviyo flow filter)

This gives you stable creative with live product truth, without the risk of sending someone to a 404 or a sold-out PDP.

```typescript
// Flow email block with hybrid resolution
type ProductCardBlock = {
  type: 'product_card'
  shopify_source: ShopifySource
  resolution: 'open_time'

  // Snapshotted at flow build — editorial, stable
  headline: string
  cta_text: string
  offer_copy?: string

  // Live at open time — Shopify Storefront API
  image_url: string       // → dynamic URL template
  title: string           // → merge tag
  price: string           // → merge tag
  product_url: string     // → merge tag

  // Inventory fallback
  oos_behavior: 'swap_related' | 'suppress' | 'show_anyway'
  fallback_product_id?: string
}
```

## Key Technical Notes for Replit

- Use `tsx` for running TypeScript server files: `"dev": "tsx watch server/index.ts"`
- Supabase client should be initialized once and exported: `/server/lib/supabase.ts`
- All Klaviyo and Shopify Admin API calls must go through the backend (never expose private keys to frontend)
- Shopify Storefront API token is safe to use from the frontend for product search
- Email HTML rendering should be server-side (keeps template logic in one place)
- Snapshot resolution (`resolveSnapshot`) runs server-side at schedule time — never in the browser
- Use TanStack Query’s `invalidateQueries` after every mutation (already partially in place)
- Add Zod schemas for all Klaviyo and Shopify API payloads before sending
- Product cache in Supabase reduces Shopify API calls during heavy editing sessions
